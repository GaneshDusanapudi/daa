### Board Representation

```python
self.board = {
    (0, 1): 'b',    # Black man
    (5, 2): 'r',    # Red man
    (3, 4): None,   # Empty dark square
}
# 'r'/'b' = men,  'R'/'B' = Kings
```

> **Panel Q: "Why a dict?"** — Only 32 of 64 squares are playable. A dict stores only these, with O(1) lookups. A 2D list wastes half its space.

### Game Rules Implemented

| Rule | Implementation |
|------|---------------|
| Movement | Men: forward diagonal. Kings: all 4 diagonals. |
| Promotion | Man reaching back row → King (`'r'` → `'R'`) |
| Forced Capture | If captures exist, simple moves are suppressed |
| Multi-Jump | Recursive chain — piece must keep jumping |

### Backtracking IN Move Generation

The multi-jump finder (`_build_capture_chain`) **itself uses backtracking** — it temporarily mutates the board to explore each jump, then undoes it:

```python
def _build_capture_chain(self, row, col, piece, path, captured_set, results):
    for each diagonal direction:
        if valid_jump_exists:
            # ── BT STEP 1: APPLY (mutate board temporarily) ──
            self.board[(row, col)] = None
            saved_mid = self.board[mid_pos]
            self.board[mid_pos] = None
            self.board[jump_dest] = piece

            # ── BT STEP 2: RECURSE ──
            self._build_capture_chain(jump_dest, ...)  # Find more jumps

            # ── BT STEP 3: UNDO (restore board) ──
            self.board[jump_dest] = None
            self.board[mid_pos] = saved_mid
            self.board[(row, col)] = piece
```

> **Panel Q: "Where is backtracking here?"** — The board is modified in-place to simulate the jump, then recursion explores further jumps. After recursion returns, the board is restored. This is classic backtracking: **explore → recurse → restore**. A double-jump returns `((6,1), (4,3), (2,5))`.

### Forced Capture Logic

```python
def generate_moves(self, player):

    if chains:
```

# PART 2 — Backtracking in AI Search (Person 2)

## The Problem: Board State Explosion

The AI explores thousands of future states. The **naive** approach copies the board at each node:

```python
# ❌ NAIVE — O(n) copy at EVERY node → O(n × b^d) total memory
new_board = copy.deepcopy(board)
new_board.make_move(move)
score = search(new_board, depth - 1)
```

## The Solution: In-Place Backtracking

```python
# ✅ BACKTRACKING — O(1) apply/undo, ONE board in memory
info = engine.apply_move(move)       # Step 1: APPLY (mutate)
score = search(engine, depth - 1)    # Step 2: RECURSE (explore)
engine.undo_move(move, info)         # Step 3: UNDO (restore)
```

### Memory Comparison

| Approach | Memory |
|----------|--------|
| Board cloning | O(n × b^d) — one copy per node |
| **Backtracking** | **O(n + d)** — one board + stack |

With b=10, d=6: cloning = ~1 million copies. Backtracking = **1 board**.

### How `apply_move` Works (BT Step 1: Apply)

```python
def apply_move(self, move):
    piece = self.board[start]
    captured_list = []

    self.board[start] = None           # Remove from origin

    for each jump leg:
        mid = midpoint(prev, dest)
        captured_list.append((mid, self.board[mid]))  # RECORD for undo
        self.board[mid] = None                        # REMOVE captured

    self.board[final] = piece          # Place at destination
    # Check promotion: 'r' at row 0 → 'R'

    return {'piece': piece, 'captured': captured_list, 'promoted': promoted}
```

### How `undo_move` Works (BT Step 3: Undo)

```python
def undo_move(self, move, info):
    self.board[final] = None                    # Clear destination
    self.board[start] = info['piece']           # Restore original piece

    for pos, captured_piece in info['captured']:
        self.board[pos] = captured_piece        # Restore ALL captured pieces

    self.turn = opponent(self.turn)
```

> **Panel Q: "How do you guarantee correctness?"** — Every mutation in `apply_move` is recorded in the `info` dict. `undo_move` reverses each mutation. After undo, the board is byte-for-byte identical to its pre-apply state.

### Backtracking Tree Visualisation

```
     search(depth=3)
     ├── apply_move(A)
     │   ├── search(depth=2)
     │   │   ├── apply_move(X) → search(1) → score
     │   │   ├── undo_move(X)              ← RESTORED
     │   │   ├── apply_move(Y) → search(1) → score
     │   │   └── undo_move(Y)              ← RESTORED
     │   └── return best
     ├── undo_move(A)                      ← RESTORED
     ├── apply_move(B) → ...
     └── undo_move(B)                      ← RESTORED
```

# PART 3 — Dynamic Programming / Memoization (Person 3)

## The Problem: Overlapping Subproblems

The same board state can be reached via different move orders:

```
Path A:  Red(5,0)→(4,1)  then  Red(5,2)→(4,3)  →  STATE X
Path B:  Red(5,2)→(4,3)  then  Red(5,0)→(4,1)  →  STATE X  (identical!)
```

Without DP, State X is evaluated **multiple times** — exponential waste.

## The Solution: Transposition Table (DP Dictionary)

### Step 1 — Generate a unique key for each board state

```python
def board_key(self) -> frozenset:
    return frozenset(
        (pos, piece) for pos, piece in self.board.items() if piece
    )
```

> **Panel Q: "Why frozenset?"** — It's **order-independent** (no sorting needed), hashable (can be a dict key), and only includes occupied squares (smaller, faster to hash).

### Step 2 — Store and lookup in the memo dictionary

```python
memo = {}   # Transposition table

def alphabeta_bt_dp(engine, depth, alpha, beta, maximizing, player):
    key = (engine.board_key(), depth, maximizing)

    # DP LOOKUP — seen this state before?
    if key in memo:
        return memo[key]['score']   # O(1) — skip entire subtree!

    # ... expensive recursive search ...

    # DP STORE — save for future
    memo[key] = {'score': best, 'flag': flag, 'depth': depth}
```

### How DP Prevents Exponential Explosion

| Without DP | With DP |
|-----------|---------|
| Same state reached 100 ways → evaluated 100 times | Evaluated **once**, cached, reused 99 times for **free** |
| O(b^d) evaluations | O(min(b^d, unique_states)) |

### The Evaluation Heuristic (What Gets Stored)

```python
def evaluate(self, player):
    score = 0
    for (row, col), piece in board:
        value = 3.0 if King else 1.0           # Material
        value += advancement_bonus(row)         # Closer to promotion
        value += 0.5  if edge_column            # Safer position  
        value += 0.3  if centre_square          # Controls more diagonals
        value += 0.2  if back_row               # Guards promotion lane
        score += value if own else -value
    return score
```

> **Panel Q: "Why not just count pieces?"** — A piece near promotion is more threatening than one on the back row. Edge pieces are safer. Centre pieces dominate. These positional weights make the AI "smarter" without searching deeper.

# PART 4 — Alpha-Beta Pruning + DP Interaction (Person 4)

## Alpha-Beta Pruning

Minimax evaluates **every** node: O(b^d). Alpha-Beta adds two bounds:
- **α (alpha)** — best the maximizer can guarantee
- **β (beta)** — best the minimizer can guarantee

**Rule**: If α ≥ β → **PRUNE** (remaining children can't affect the result).

```python
for move in moves:
    info = engine.apply_move(move)         # ← Backtracking
    val = alphabeta(engine, depth-1, alpha, beta, ...)
    engine.undo_move(move, info)           # ← Backtracking

    if maximizing:
        best = max(best, val)
        alpha = max(alpha, val)            # Raise floor
    else:
        best = min(best, val)
        beta = min(beta, val)              # Lower ceiling

    if alpha >= beta:
        break   # ← PRUNE! Skip remaining siblings
```

### Pruning Example

```
            MAX (has 5 from left branch)
           / | \
         5   3   ?
              ↑
         MIN found 3 here
         MAX already has 5 → will NEVER pick ≤3
         → PRUNE "?" — don't even evaluate it
```

### How DP Flags Enable More Pruning

When Alpha-Beta prunes, the score isn't exact. We store the **type**:

| Flag | Meaning | When |
|------|---------|------|
| `EXACT` | Score is exact | No pruning at this node |
| `LOWER_BOUND` | True score ≥ stored | Beta cut-off |
| `UPPER_BOUND` | True score ≤ stored | Alpha cut-off |

On future lookups, flags narrow the α-β window:

```python
if entry['flag'] == EXACT:
    return entry['score']              # Use directly
elif entry['flag'] == LOWER_BOUND:
    alpha = max(alpha, entry['score']) # Raise floor
elif entry['flag'] == UPPER_BOUND:
    beta = min(beta, entry['score'])   # Lower ceiling
if alpha >= beta:
    return entry['score']              # Window collapsed → prune
```

> **Panel Q: "How do BT + DP + Alpha-Beta all work together?"**

```
alphabeta_bt_dp():
  ├─ DP LOOKUP:  board_key in memo? → return cached / narrow window
  ├─ For each move:
  │   ├─ apply_move()           ← BACKTRACKING
  │   ├─ recurse(α, β)         ← ALPHA-BETA window
  │   ├─ undo_move()           ← BACKTRACKING
  │   └─ if α ≥ β → PRUNE     ← skip remaining
  └─ DP STORE: memo[key] = {score, flag, depth}
```

### Complexity Analysis

**Time:**

| Algorithm | Nodes | At d=6, b=10 |
|-----------|-------|---------------|
| Plain Minimax | O(b^d) | 1,000,000 |
| Alpha-Beta | O(b^(d/2)) | ~1,000 |
| **Alpha-Beta + DP** | Even fewer | **~2,800 actual** |

**Space:**

| Component | Space |
|-----------|-------|
| Board (BT — single instance) | O(n), n=32 |
| Recursion stack (BT) | O(d), d=6 |
| Transposition table (DP) | O(\|S\|) unique states |
| **Total** | **O(n + d + \|S\|)** |

> Without BT, space would be O(n × b^d) for board copies. BT reduces to O(n + d).
